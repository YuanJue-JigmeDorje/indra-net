# SKILL 04a: 事件识别 — 从标注文本到结构化事件记录

> 从敦珠《藏密佛教史》第三品 PoC 中提炼，适用于藏传佛教汉译典籍的事件抽取。

---

## 一、概述

**输入**：`source/<book>/chapter_md/<NN>_<title>.md` + `kg/entities/<chapter_id>.yaml`
**输出**：`kg/events/<chapter_id>.yaml` 事件记录
**何时使用**：实体标注（SKILL_03a）完成之后、关系构建（SKILL_05a）之前。

**核心目标**：从文本中识别所有可结构化的事件，标注参与者及其角色、时空信息、叙述层，输出 yaml。

**事件 vs 关系**：事件是一次性的、有时间锚点的发生（"松赞干布建造大昭寺"）；关系是持久的状态（"松赞干布是观音化身"）。关系在 SKILL_05a 处理。

## 三、关键规则

| # | 规则 | 说明 |
|---|---|---|
| 1 | 多角色事件 = 一个 record | 出家仪式有亲教师/羯磨师/屏教师/受戒者 → 一条事件，participants 列出各角色（原则 #20） |
| 2 | 通称非人写 description | "凶神恶煞""鬼神"等无名非人不引用 entity_id，在 description 中文字提及 |
| 3 | 物体可作主体 | "佛像开口说话" → 佛像是 participant.subject（原则 #19） |
| 4 | 授记双重身份 | 授记既是叙述层标签（该断言来自预言），也是事件类型（发出预言的行为）（原则 #18） |
| 5 | 叙述层必填 | 每个事件至少一个 layer（史传/典据/授记/宗派/神变/证境/传闻），参照 narrative-layers v0.2 |
| 6 | 不猜时间 | 文本未给出时间信息则 time_info 留空（原则 #12） |
| 7 | 同一事件多说法 | 用多条 record 各带不同 layer + sub_layer 区分异说（如莲师驻藏时长的短/中/长观点） |

## 四、Agent Prompt

以下是事件抽取 Agent 的完整 prompt。

```
你是一个藏传佛教典籍事件抽取 Agent。

## 任务
阅读以下源文本，参照实体注册表，提取所有可结构化的事件，输出 yaml。

## 实体注册表
{ENTITIES_YAML}

## 事件类型（可扩展）
迎请 / 出家受戒 / 建寺 / 灌顶 / 译经 / 降伏非人 / 授记 / 离藏 / 弘法 / 传法 /
伏藏埋藏 / 伏藏开取 / 毁灭佛教 / 恢复佛教 / 执政 / 圆寂 / 闭关修行 / 派遣 /
创制文字 / 开光 / 立法 / 神变

如遇不属于以上类型的事件，用最贴切的新类型名，在 note 字段说明。

## 关键规则
- 多角色事件记为一条 record，participants 分列各角色
- 通称非人（"鬼神""凶神恶煞"）写在 description 中，不引用 entity_id
- 佛像/法器等 object entity 可以作为 participant.subject
- 授记：同时标 layer: [授记] 和 type: 授记
- 每个事件必须有 layer（至少一个叙述层）
- 文本没说的时间留空，不猜
- 多种说法的同一事件 → 多条 record，各带不同 layer/sub_layer
- confidence 默认值：史传 0.85 / 典据 0.95 / 授记 0.95 / 神变 0.75 / 传闻 0.60

## 输出格式
每个事件一条 yaml record：
```yaml
EVT_NNN:
  type: 事件类型
  description: 一句话描述（中文）
  participants:
    - entity: E_NNN          # 引用实体注册表 id
      role: subject           # subject / agent / recipient / location_host / ...
    - entity: E_NNN
      role: 亲教师            # 佛教特定角色可用中文
  location: E_NNN             # 地点 entity_id（可选）
  time_info:
    text: "十三岁时"          # 原文时间表述
    relative_to: E_NNN        # 相对于谁（如"赤松德赞十三岁"）
    absolute: null             # ISO 日期（极少有，通常 null）
  layers: [史传]              # 叙述层标签列表
  sub_layer: null              # 异说区分（如"短时间观点"）
  author_stance: 中立          # 认可/中立/怀疑/反驳
  confidence: 0.85
  source_quote: "原文引用片段"
  note: null
```\n

## 源文本
{TEXT}
```

## 五、执行流程

```
1. 加载实体注册表 kg/entities/<chapter_id>.yaml
2. 通读源文本，逐段识别事件
3. 对每个事件：
   a. 判定事件类型（§二 类型表）
   b. 识别所有参与者 → 匹配 entity_id + 角色
      - 多角色事件 → 一条 record，多个 participants
      - 通称非人 → description 中提及，不建 participant
   c. 识别地点 → 匹配 location entity_id
   d. 提取时间信息 → time_info（原文表述 + 相对锚点）
   e. 判定叙述层 → layers[]（参照 narrative-layers v0.2）
   f. 判定 author_stance → 默认中立
   g. 设定 confidence → 按叙述层默认值，视上下文微调
   h. 提取 source_quote → 直接引用原文关键语句
4. 检查：同一事件是否有多种说法 → 拆为多条 record
5. 输出 yaml 到 kg/events/<chapter_id>.yaml
6. 运行质量检查（§六）
```

## 六、质量检查

| # | 检查 | 方法 |
|---|---|---|
| 1 | **无类型事件** | 所有 record 的 type 字段非空 |
| 2 | **无参与者事件** | 所有 record 至少有一个 participant |
| 3 | **entity_id 有效** | participants 中的 entity id 都能在实体注册表中找到 |
| 4 | **layer 必填** | 所有 record 的 layers 列表非空 |
| 5 | **通称非人泄漏** | participants 中不应出现通称非人的 entity_id |
| 6 | **重复事件** | 检查是否有 description 高度相似的 record → 合并或标注为异说 |
| 7 | **授记双标** | type=授记 的事件是否同时包含 layer=授记 |
| 8 | **遗漏扫描** | 回扫全文，检查是否有明显的动作/事件未被抽取 |

## 七、输出位置与命名

```
kg/
├── entities/
│   └── ch03.yaml
├── events/
│   ├── ch03.yaml                # 第三品全章事件
│   └── ...
└── relations/                   # SKILL_05a 输出
```

全章 vs 子节：首次按全章出一个文件；如果事件数 > 100，拆成子节文件。

## 八、已知局限

1. **事件粒度主观**：一段叙事可拆成一个大事件或多个小事件，Agent 偏向有意义的最小单位
2. **时间信息稀疏**：第三品多用相对时间（"十三岁时""此后第五代"），绝对年份极少
3. **通称非人丢失**：大量无名鬼神的参与只能以文字描述捕捉，不进结构化字段
4. **因果链不在本阶段**：事件之间的因果/先后关系在 SKILL_07 处理
5. **跨章事件**：同一事件跨章提及时，以首次出现的章节为主记录，后续章节用引用
